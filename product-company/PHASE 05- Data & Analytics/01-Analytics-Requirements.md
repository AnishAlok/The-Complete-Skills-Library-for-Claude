# analytics-requirements.md

> **Skill:** Produces analytics instrumentation requirements that define what user actions, system events, and business outcomes must be tracked before a feature launches.

---

## TLDR

This skill helps you answer: *"What instrumentation must be in place before this feature ships so that we can measure whether it is working?"* Analytics requirements define the events, properties, user properties, and business metrics that must be tracked for a feature. Written before development begins so that instrumentation is built into the feature, not retrofitted after launch.

---

## Topic Explanation

### Why Analytics Requirements Belong in the Development Process

The most common analytics failure mode: a feature ships and the team discovers it has no usable data to measure success. Retrofitting instrumentation is expensive: it requires another engineering cycle, and the data gap between launch and instrumentation means the team cannot answer the most important questions about the feature's initial adoption.

Analytics requirements written alongside the PRD ensure that instrumentation is treated as a first-class engineering requirement, not an afterthought.

### The Three Levels of Analytics Instrumentation

**Product events:** User actions within the product. Button clicks, page views, form submissions, feature interactions. Captured by the analytics SDK (Segment, Mixpanel, Amplitude).

**System events:** Backend events generated by the application. Job completions, API calls, errors, state changes. Captured by server-side instrumentation.

**Business outcomes:** Conversions, revenue events, subscription changes, support tickets. May span multiple systems. Often require combining product and business data.

### Event Properties vs. User Properties

**Event properties:** Data recorded with a specific event. "Clicked Export" → {report_type: "compliance", date_range: "30d", format: "pdf"}.

**User properties:** Attributes of the user recorded in their profile and available for segmentation. plan_type, company_size, signup_date, lifecycle_stage.

Event properties answer "what happened in this instance." User properties answer "what kind of user did this."

### The Tracking Plan Relationship

Analytics requirements define *what* must be tracked and *why*. The tracking plan (see `tracking-plan.md`) defines the exact technical specification: event names, property names, data types, and validation rules. Requirements come first; the tracking plan implements them.

## When to Use This Skill

Use when:
- A new feature is entering development and measurement must be specified
- An existing feature has insufficient data to answer key product questions
- A major product area is being redesigned and instrumentation needs to be updated
- A data team is auditing what is and is not tracked in the product

---

## Dos

- **Do write analytics requirements before development begins.** Retrofitting is more expensive than building it in.
- **Do specify the question each event answers.** Every event should exist because it answers a specific question. "We track X because it tells us Y."
- **Do include error states and edge cases.** What should be tracked when something fails? When a user abandons mid-flow?
- **Do specify the business questions the data must answer.** Start from the questions, derive the events.
- **Do include privacy requirements.** Which events involve PII? What must not be tracked?

---

## Donts

- **Dont track everything because you can.** Over-instrumentation creates noise, increases storage costs, and makes the data harder to use. Track what answers specific questions.
- **Dont specify implementation in analytics requirements.** Requirements say what must be tracked. The tracking plan says exactly how.
- **Dont skip the "why" for each event.** Events without documented purpose get orphaned and no one knows if they are safe to change.

## Output Format

```
1. REQUIREMENTS HEADER
   Feature, author, date, PRD reference, status.
   Analytics lead, engineering implementer.

2. MEASUREMENT OBJECTIVES
   The business and product questions this instrumentation must answer.
   How these connect to the feature's success criteria.
   The decisions this data will enable.

3. SUCCESS METRICS (what we are measuring for)
   Primary success metric.
   Secondary metrics.
   Guardrail metrics (must not degrade).
   Counter-metrics.

4. USER JOURNEY TO INSTRUMENT
   The key user flow with instrumentation points marked.
   Entry point → key interactions → success event → exit/abandonment.

5. REQUIRED EVENTS

   For each event:
   - Event name (verb + noun format: "Clicked Export")
   - Trigger: exactly when this fires
   - Why: what question this event answers
   - Required properties: name / type / example value / description
   - Optional properties
   - Volume estimate: expected fires per user per day

6. REQUIRED USER PROPERTIES
   Properties that must be set/updated as part of this feature.
   Name / type / when set / description.

7. FUNNEL DEFINITION
   The ordered sequence of events that constitute the key funnel.
   Expected drop-off rates for each step.
   What constitutes funnel completion.

8. ERROR AND ABANDONMENT TRACKING
   Errors that must be tracked.
   Abandonment points to capture.

9. PRIVACY REQUIREMENTS
   Events that involve PII: what must not be tracked.
   Consent requirements.
   Data retention limits.

10. INSTRUMENTATION ACCEPTANCE CRITERIA
    How the PM/analyst will verify instrumentation is correct before launch.
    Test cases: specific user actions and expected event output.
    QA checklist for instrumentation review.
```

## Starter Prompt

```
You are a product manager or analyst writing analytics instrumentation requirements.

Feature: [what is being built]
PRD reference: [link to product requirements]
Business questions to answer: [the key questions the data must enable]
Success metrics: [how the feature's success will be measured]
Key user flows: [the important paths through the feature]
Known privacy constraints: [any PII or consent considerations]

Please produce complete analytics requirements including:
1. Measurement objectives connected to success criteria
2. Required events with trigger conditions, properties, and "why" for each
3. Required user properties
4. Funnel definition with expected conversion rates
5. Error and abandonment tracking
6. Privacy requirements
7. Instrumentation acceptance criteria

Flag any success metric that cannot be measured with the proposed events.
Flag any event that tracks PII without a documented privacy justification.
```
